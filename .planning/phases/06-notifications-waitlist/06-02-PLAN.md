**Create `mounttrack/src/lib/notifications.ts`:**

Initialize Twilio and Resend clients at module level (singleton). Export three functions:

`sendStageNotification(payload)` — sends SMS + email on stage change, persists to notifications table.
`sendPaymentRequestNotification(payload)` — sends SMS + email with payment link.
`sendWaitlistSms(payload)` — sends SMS only for waitlist confirmation (no email per decisions).

All three functions follow the same pattern:
- SMS: skip silently if customerPhone is null OR sms_opted_out is true. Catch errors, log, never rethrow.
- Email: skip silently if customerEmail is null. Catch errors, log, never rethrow.
- After successful sends, insert row(s) into notifications table. Wrap in try/catch — persistence failure must NOT propagate.

Payload interface for sendStageNotification:
```typescript
interface StageNotificationPayload {
  customerPhone: string | null
  customerEmail: string | null
  smsOptedOut: boolean
  customerName: string
  stageName: string
  portalUrl: string
  shopName: string
  shopLogoUrl: string | null
  brandColor: string
  jobId: string
  shopId: string
}
```

SMS body format (per user decision): `"${shopName}: Great news! Your mount has moved to ${stageName}. Track progress: ${portalUrl}"`

Resend `from` field: `"${shopName} <notifications@mounttrack.app>"` — executor may use a placeholder; actual domain verification is a deploy prerequisite.

sendPaymentRequestNotification payload: same fields but stageName is not needed; SMS body: `"${shopName}: Your mount is ready! Complete your payment here: ${portalUrl}"`. Email subject: "Your mount is ready — complete payment".

sendWaitlistSms payload: `{ customerPhone, shopName, animalType, shopId }`. SMS body (per user decision): `"${shopName}: You're on our waitlist! We'll reach out when we're ready for your ${animalType}."`. Persists to notifications table with type='waitlist_confirm', job_id=null, channel='sms'.

Use createServiceClient (from @/lib/supabase/service) for notifications table inserts — this module is called from server actions which have their own client, but using service client in the helper avoids passing the caller's client through. Import createServiceClient once at top.

Create the new file `mounttrack/src/app/api/webhooks/twilio/route.ts`.

Key facts:
- Twilio sends `application/x-www-form-urlencoded` — use `req.formData()` NOT `req.json()`
- `OptOutType` field is 'STOP' when customer sends a stop keyword
- `From` field is customer's E.164 phone number
- `To` field is the Twilio number that received the message — use this to scope the shop lookup
- Use `createServiceClient()` — no auth session in webhook context
- The path `/api/webhooks/twilio` is already excluded from auth middleware (same pattern as stripe webhook)
- Update ALL jobs matching customer phone + shop_id. In v1, one Twilio number = one shop. Map via TWILIO_PHONE_NUMBER env var if multiple shops exist (future concern), but for v1 update by customer_phone alone is acceptable per research pitfall 2 note. However, add `.eq('shop_id', shopId)` for correctness — to find shopId, we need to look up which shop owns the Twilio number. Since v1 has one shop per Twilio account, a simpler approach: update all jobs with matching customer_phone regardless of shop (acceptable for v1 per research). Document this limitation.

```typescript
import { createServiceClient } from '@/lib/supabase/service'
import { NextResponse } from 'next/server'

export const runtime = 'nodejs'

export async function POST(req: Request) {
  const formData = await req.formData()
  const from = formData.get('From') as string | null      // customer phone E.164
  const optOutType = formData.get('OptOutType') as string | null  // 'STOP' | 'START' | 'HELP' | null

  // Only act on STOP opt-outs
  if (optOutType !== 'STOP' || !from) {
    return NextResponse.json({ received: true })
  }

  try {
    const supabase = createServiceClient()
    // Set sms_opted_out on all jobs for this customer phone number.
    // V1: one Twilio account = one shop; no cross-shop risk.
    // eslint-disable-next-line @typescript-eslint/no-explicit-any
    await (supabase.from('jobs') as any)
      .update({ sms_opted_out: true })
      .eq('customer_phone', from)
  } catch (err) {
    console.error('[twilio-webhook] failed to set sms_opted_out:', err)
    // Return 200 regardless — Twilio retries on non-2xx; logging is sufficient
  }

  return NextResponse.json({ received: true })
}
```
